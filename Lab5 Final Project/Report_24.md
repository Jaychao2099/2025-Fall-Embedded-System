## Part 1 即時物件辨識 (Real-Time Object Recognition)

### 1. 系統概述

*  **目標**：在嵌入式平台 (E9V3) 上實現即時物件偵測的系統架構與關鍵優化技術。
*  **實作方法**：我們採用輕量化的 YOLOv8n 模型搭配 TNN 推論引擎，並針對 ARM 架構進行了底層優化以達到即時性要求。

### 2. 模型選擇與訓練

為了在嵌入式裝置上達到最佳的推論速度，我們選擇了 **YOLOv8-nano** 模型。

* **輸入尺寸：** 將模型輸入尺寸調整為 **224x224**，大幅降低運算量以符合即時需求。
* **資料集：** 使用 Roboflow 已存在的 datasets 訓練題目指定的 8 類物件 (banana, book, bottle, cell phone, cup, keyboard, scissors, spoon)。
* **模型轉換：** 訓練完成的 PyTorch 模型先轉為 ONNX 格式，再進一步轉換為 **TNN (Tencent NCNN)** 格式 (`.tnnproto`, `.tnnmodel`)，以利用 TNN 對 ARM CPU 的指令集優化。

### 3. 軟體架構：多執行緒平行處理

為了避免推論過程阻塞視訊畫面的更新，在 C++ 主程式採用了非同步的 **Producer-Consumer** 架構：

* **主執行緒 (Main Thread)：** 負責 Camera 取像、繪製 Bounding Box 以及驅動 Framebuffer 顯示。確保畫面流暢，不受後端運算影響。
* **AI 工作執行緒 (AI Worker Thread)：** 獨立運行的執行緒，持續從共享緩衝區讀取最新影像進行推論。
* **同步機制：** 使用 `std::mutex` 與 `std::atomic` 變數保護共享資料 (`global_objects`, `ai_buffer`)，確保資料讀寫的安全性與即時性。

### 4. 關鍵效能優化

為了在有限的硬體資源下極大化 FPS，我們實作了以下優化：

* **ARM NEON SIMD 加速 NMS：**
在後處理階段，我們手動實作了基於 **ARM NEON Intrinsics** 的非極大值抑制 (NMS) 演算法。透過 SIMD 指令 (如 `vmaxq_f32`, `vmulq_f32`) 一次平行處理 4 個候選框的 IoU 計算，期望減少 CPU 迴圈的耗時。
* **直接 Framebuffer 存取：**
不使用高階視窗系統，而是直接透過 `mmap` 映射 `/dev/fb0` 裝置。將 OpenCV 的影像資料直接轉換為 **BGR565** 格式並寫入記憶體，大幅降低顯示延遲。
* **記憶體管理優化：**
預先配置固定的記憶體緩衝區 (`ai_buffer`)，避免在每一幀處理中重複進行 `malloc/free`，減少系統開銷。此外，解析 TNN 輸出時直接計算 NC4HW4 記憶體偏移量，避免不必要的資料複製。
* **CPU 效能模式：**
透過程式自動將 CPU governor 設定為 `performance` 模式，確保處理器全速運行。

### 5. 流程總結

1. **取像：** 從攝影機讀取 640x480 影像。
2. **預處理：** Resize 至 224x224，並進行 Normalize 與格式轉換 (BGR to RGB, HWC to NCHW)。
3. **推論：** TNN 引擎執行 Forward Pass。
4. **後處理：** 解析 Output Blob，執行信心度篩選與 NEON 加速的 NMS，將座標還原回 640x480。
5. **顯示：** 將結果疊加於原圖並輸出至 HDMI/LCD。

## Part 2 高解析度物品辨識 (high-resolution Object Recognition)

### 1. 系統概述
與 part 1 相同，我們採用輕量級的 YOLOv8n 模型，並搭配 TNN 推論框架進行加速。為了在低解析度推論下仍能保有對微小物件的偵測能力，導入了「切圖推論 (Tiling Inference)」策略，並針對嵌入式環境優化了 NMS (非極大值抑制) 演算法。

### 2. 系統架構與流程
整體系統流程分為「模型訓練」與「嵌入式推論」兩大階段。

#### 2.1 模型選擇與訓練
*	**模型架構**：選用 YOLOv8-Nano，因其參數少、運算量低，最適合嵌入式裝置。
*	**資料集準備**：使用 Roboflow 對題目指定的 40 類物件進行資料集管理，針對特定類別進行資料增強 (Augmentation)，包括 Mosaic 拼貼與 HSV 色調調整，以提升模型對光影變化的穩健性。
*	**訓練策略**：
    *	**Input Resolution**：設定為 320x320，為了配合開發板推論速度所做的關鍵權衡。
    *	**Training Epochs**：採用 Early Stopping 策略，並在最後 10 Epochs 關閉 Mosaic 增強，讓模型收斂於真實圖片特徵。

#### 2.2 模型轉換與部署
*	**轉換流程**：把 ONNX 轉換成TNN。
*	**精度設定**：
    *	第一階段轉換關閉半精度以保留 FP32 精度。
    *	推論引擎設定為 `PRECISION_AUTO`，讓 TNN 自動適應 ARM 指令集優化。

### 3. 程式實作細節
本系統也使用 C++ 開發，核心功能分為圖像預處理、TNN 推論引擎、與後處理邏輯。

#### 3.1 切圖策略 (Tiling)
為了在 320x320 的低解析度輸入下，偵測高解析度原圖中的微小物件，我們實作了滑動視窗切圖：
*	**動態切圖**：將原始大圖切割為多個 320x320 的子圖 (Tiles)。
*	**Overlap**：設定重疊區域 (如 60~80 pixels) 以防止物件因位於切割邊界而被截斷或漏檢。
*	**座標還原**：推論後的子圖座標會加上 Offset 還原回原圖的全域座標。

#### 3.2 圖像預處理 (Preprocessing)
實作了 Letterbox 演算法：
*	**目的**：確保模型輸入不會因變形而影響辨識準確度。
*	**方法**：保持長寬比縮放，並在不足處填充灰色邊框 (Pad 114)。

#### 3.3 後處理優化 (Post-Processing)
為了過濾重複框與提升準確率，我們對標準 NMS 進行了改良：
1.	**無視類別 NMS (Class-Agnostic NMS)**：
    *	**問題**：模型有時會對同一物件預測出兩個不同類別的框 (如同時判定為 Cat 與 Dog)。
    *	**解法**：移除 NMS 中的類別檢查，只要位置重疊率過高，即使類別不同也強制合併，優先保留信心度高者。
2.	**暴力 NMS 參數**：
    *	採用極低的 IoU 閾值 (`nms_thres` = 0.15)，以消除切圖產生的邊緣鬼影。
3.	**IOU_MIN 計算策略**：
    *	針對「大框包小框」的重疊情況，將 IoU 計算的分母改為 `min(area1, area2)`，有效移除非必要的包含型重疊框。

### 4. 顯示與輸出
*	**Framebuffer 直接寫入**：利用 Linux Framebuffer 驅動，透過 mmap 將處理後的影像直接寫入視訊記憶體，實現無 X-Window 環境下的高效顯示。
*	**OpenCV 繪圖**：在記憶體中完成畫框與標註文字後，再一次性輸出的螢幕。

### 5. 遭遇困難與解決方案
|遭遇困難|解決方案|
|:---:|---|
|**Timeout**|	將模型輸入由預設 640 改為 320，運算量大幅降低 75%，成功在時限內完成。
|**Classes Overlapping**|	實作 Global NMS 並移除類別判斷，並大幅調低 IoU Threshold 至 0.15。|
|**Miss detection**|	透過調整 `conf_thres` 至極低 (0.15) 讓模型更敏感，再靠嚴格的 NMS 過濾雜訊。|

## 總結
本次實驗成功在資源受限的嵌入式平台 (E9V3) 上整合 **YOLOv8-nano** 模型與 **TNN 推論引擎**，實現高效能物件辨識。

*   針對 **即時串流**，我們採用 **Multi-threading** 架構分離 I/O 與運算，並結合 **ARM NEON SIMD 指令集** 優化 NMS 及 **Framebuffer** 直接記憶體存取，大幅降低延遲。
*   針對 **高解析度偵測**，導入 **Tiling** 以捕捉微小物件，並透過 **Class-Agnostic NMS** 與 **Aggressive IoU 參數調校** 有效過濾重疊誤判。

整體透過 **模型輸入降維** (224/320 px) 與 **底層記憶體管理**，在有限算力下達成了即時性與準確度的最佳平衡。
